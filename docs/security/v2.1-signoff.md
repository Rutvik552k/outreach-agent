# Security Re-Review — V2 Actor-Binding Amendment (ADR v2.1, contract C4)

**Reviewer:** security-engineer | **Scope:** AC3 confused-deputy closure under the v2.1 layered replacement | **Method:** design reasoning over ADR v2.1 + original threat model. No web research.

## Threat Model Summary (recap, unchanged)

- **Asset:** A4 user reputation / A1 token-as-user — nothing reaches upstream without a *human* approval.
- **Trust boundary crossed by AC3:** the agent's own `public_repo` token has triage on the user's fork (S1+S2), so it is *capable* of applying `agent:approve-upstream` / posting `/approve`. The confused-deputy is the agent self-approving (prompt-injection or bug) and bypassing the human gate (FR-3, the product's core safety invariant).
- **What changed:** my original V2 hard rule `actor.login != agent_oauth_login` is confirmed structurally unsatisfiable — the OAuth user-to-server token acts AS the user (ADR §4), so timeline label-add / comment events carry the *user's own* login regardless of whether a human in a browser or the agent via API performed them. Login comparison provably cannot distinguish the two. The implementer is correct; my original control was wrong.

## Q1 — Does the layered replacement close AC3 at the intended assurance level?

**Yes, and arguably at a *higher* assurance level than my original rule — conditional on the structural-incapability layer being real, not aspirational.**

Reasoning, decomposed by why the original rule existed vs. what replaces it:

- My original V2 was a **detective control at the actor layer** (compare who acted). It was the only lever I saw at design time. It is now proven non-functional — a detective control that cannot discriminate is zero assurance, not partial. So v2.1 is not weakening a working control; it is replacing a broken one.
- v2.1 layer (2) **Structural incapability** is a **preventive control at the capability layer**: the agent cannot emit the approval signal at all because no GitHubGateway method (C5) constructs an approval-label-add or `/approve` comment on an existing fork draft PR, and the generic `comment` mutation is restricted away from that target. This is strictly stronger than detection: you cannot self-approve through a surface that does not exist. **Critically, this layer's assurance depends entirely on the gateway being the *only* path to the GitHub HTTP client.** The ADR asserts this is enforced by (a) "no such gateway method exists (asserted by test)" and (b) the existing build-breaking lint rule "no HTTP client use outside C5" (C4, C5, C7). That lint rule is load-bearing — it is the thing that makes "structural incapability" true rather than "incapable via the happy path." See Q2.
- The fact you raise — **the agent is deterministic Python, the LLM only produces text/diffs, never actions** — materially *raises* the assurance of layer (2). The LLM cannot call the gateway; it cannot synthesize an HTTP request; its output is data (patch, PR body, review-reply draft) that flows into human-gated queues. So the prompt-injection path to self-approval (AC3's scariest variant, AC1-adjacent) requires the injected text to cause the *deterministic Python* to do something it has no code path to do. Injection can poison the **diff content** (still B2/AC1, handled by V5 diff-review hardening) but cannot reach the **approval-signal-emission capability** because that capability is absent from the codebase. This is a clean architectural separation: text/diff plane vs. mutation-capability plane.
- v2.1 layer (3) **Audit cross-check at the atomic pre-publish gate** is the **detective backstop** for the residual case where layer (2)'s assumption fails (a bug emits a mutation that *does* touch the draft PR's labels/comments). Because every gateway mutation emits hash-chained intent/confirmed events (C5/C6), an agent-originated label/comment on the draft PR leaves an audit trace; the gate rules that signal ineligible. Hash-chain (V4) makes the trace tamper-evident.

So the defense is: **prevent (can't emit) → detect (if emitted, it's audited and rejected) → document residual (full host compromise out of scope).** That is genuine defense-in-depth and closes AC3 at intended assurance **provided** the structural layer's "only path is C5" invariant holds and the audit cross-check matching rule is unambiguous (Q3). It is a stronger posture than the single broken detective control it replaces.

## Q3 — Is the audit cross-check well-defined enough to implement?

**No — it is currently under-specified, and the ambiguity is security-relevant.** C4 says: "if any agent mutation (intent/confirmed) targets the draft PR's labels or comments in the approval window, the corresponding events are ineligible." Three undefined terms:

1. **What is "the approval window"?** Undefined. If it's "since draft creation," any benign agent comment on the draft (e.g., a status note) could nuke a *legitimate* human approval (false-positive → DoS on approval). If it's too narrow, an agent self-label just outside the window slips through (false-negative → bypass). **Required:** define the window as **[draft-PR-open-event, pre-publish-gate-evaluation]** — the entire lifetime of the draft as an approval surface. A legitimate human approval and an agent label-mutation cannot coexist on the same draft without the agent mutation being a violation, because **the agent has no legitimate reason to ever mutate the draft PR's labels or post `/approve`-class comments** (layer 2 guarantees it never does). So the rule simplifies: *any* agent-originated label/comment event on the draft PR's approval-relevant surface, at any time in the draft's life, makes the draft ineligible — there is no benign agent label/comment on the draft to confuse it with.

2. **What is the matching key — event-id, timestamp window, or endpoint+target?** This is the crux and C4 leaves it open. My required rule, in precedence order:

   - **Primary key — GitHub event id.** The approval signal comes from the timeline/events API (C4 owner-check already reads `actor.login` from there) and carries a GitHub event id. The audit log (C6) records `endpoint` and `contribution_id` but **does not currently store the GitHub-returned event/label id of the mutation it performed.** **Required change to C6:** when the gateway performs any label/comment mutation, capture and store the **GitHub object id returned by the mutation** (the label-event id / comment id) in the `confirmed` audit event. Then the cross-check is an **exact id-set membership test**: `approval_signal.event_id ∈ {agent_confirmed_label_or_comment_ids on this draft}` → ineligible. Exact, no clock dependency, no window fuzz.
   - **Fallback key — endpoint + target + actor=agent.** If the GitHub event id of a label-add is not reliably retrievable to correlate (timeline label events vs. the issues-label endpoint response may not share an id — **this is itself UNVERIFIED and must be confirmed at implementation**), fall back to: *the draft PR has ANY `actor==agent, phase==confirmed` audit event whose `endpoint` matches a label-add or comment-create template targeting this draft's number* → the draft is ineligible wholesale (not the specific signal — the whole draft). This is coarser but **fails safe** (rejects approval, never publishes) and needs no timestamp matching.
   - **Reject timestamp-window matching as the primary key.** Clock-based correlation (match if agent-mutation-ts within N seconds of signal-ts) is the worst option: it has both false-positive (benign coincidence) and false-negative (clock skew, slow propagation) failure modes and is gameable. Do not use it.

3. **Tie-break / fail-direction must be specified:** on *any* ambiguity in matching, the gate **aborts the publish** (fail-closed), audits the abort, and alerts — never publishes on an unresolved match. The ADR's gate already aborts on check failure (§6, C4 step), but the *cross-check's own ambiguity* must inherit that fail-closed posture explicitly.

**Required matching rule (the one I'll sign off against):** Store the GitHub-returned mutation object id in the `confirmed` audit event (C6 change). Cross-check = exact id-set membership of the approval signal's event id against agent-confirmed label/comment ids on the draft, over the draft's full lifetime. If id-correlation is unverifiable at implementation, fall back to coarse rule (*any* agent label/comment mutation on the draft ⇒ whole draft ineligible). Any ambiguity ⇒ fail-closed abort.

## Q4 — Verdict

**SIGN-OFF-WITH-CONDITIONS.**

The v2.1 amendment is architecturally sound and a genuine improvement: it replaces a provably broken detective control with a prevent→detect→document-residual stack, and the deterministic-Python / text-only-LLM separation makes the highest-likelihood attack paths (P1 bug, P3 injection) structurally impossible rather than merely audited. I am satisfied AC3's confused-deputy threat is closed at the intended assurance level **once the following conditions are met.** None reopen the architecture; all are precise and testable.

**Conditions (all must be satisfied before implementation sign-off converts to final):**

- **C-1 [BLOCKER] — The "no HTTP client outside C5" lint rule is the load-bearing control for layer (2); it must be CI-enforced and build-breaking, with a test proving a violating import fails the build.** Structural incapability is only real if the gateway is provably the sole path to the GitHub client. This must be verified to exist and bite, not assumed. (Closes P2's prevent-side.) Add a positive test: an attempted raw label-add anywhere outside C5 fails CI.

- **C-2 [BLOCKER] — Specify the audit cross-check matching rule per Q3:** store the GitHub-returned mutation object id in the C6 `confirmed` event; cross-check by exact id-set membership over the draft's full lifetime; documented coarse fallback (any agent label/comment mutation on the draft ⇒ whole draft ineligible) if id-correlation is UNVERIFIED at implementation; any ambiguity ⇒ fail-closed abort. The current C4 wording ("matched against the audit log … in the approval window") is too loose to implement unambiguously.

- **C-3 [SHOULD-FIX, HIGH] — Add the gateway-bypass residual (P2) to the documented residual-risk set, alongside full-host-compromise (P5).** Currently layer (4) documents only full host compromise. A gateway-bypassing mutation *also skips the audit wrapper*, so the cross-check (layer 3) cannot catch it — the only thing standing between P2 and a forged signal is the C-1 lint rule at build time. That dependency must be stated, not implied.

- **C-4 [SHOULD-FIX, MEDIUM] — State the single-instance assumption explicitly (P4-second-instance).** The audit cross-check's completeness assumes exactly one agent instance / one hash chain per token. Document that running a second agent instance with the same OAuth token defeats the cross-check (its mutations are in a different chain). Consistent with the single-user local trust model, but it is an unstated assumption today.

- **C-5 [SHOULD-FIX, MEDIUM] — The "in-process malicious dependency" residual (P5) is accepted as out-of-scope ONLY if the agent's own supply chain is pinned and integrity-verified.** The ADR mandates hash-verification for downloaded *model weights* and sandboxing for *repo* code, but does not mandate lockfile-pinning + hash verification for the *agent's own venv*. Since an in-process malicious dep has the token and can forge both the API call and the audit chain head (the exact residual layer 4 documents), that residual is only legitimately out-of-scope if the agent's deps are locked and hash-checked. Recommend a lockfile with hashes for the agent itself.

**Not blocking:** The owner-check (layer 1, `actor == fork_owner`) and the atomic pre-publish gate (§6) are sound and unchanged — keep. The residual full-host-compromise carve-out (layer 4) is consistent with V4's tamper-*evidence* (not prevention) posture and is correctly out-of-scope for single-user local.

Once C-1 and C-2 (the two blockers) are reflected in the ADR/implementation and C-3..C-5 are documented, this converts to full **SIGN-OFF**.

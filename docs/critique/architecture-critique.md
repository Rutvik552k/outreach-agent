# Architecture Critique — ADR-001 Outreach Agent (SDLC Phase-4 Gate)

- **Date:** 2026-06-11
- **Critic:** testing-engineer (carrying qa-engineer functional-risk hat for this gate)
- **Target:** `docs/adr/ADR-001-outreach-agent-architecture.md`
- **Against:** `docs/requirements.md` v0.2 (AC 1–7), `docs/delivery-plan.md` v0.2 invariants, `CLAUDE.md` Rule 1
- **Method:** adversarial, evidence-backed. Every finding cites ground truth (primary GitHub docs, SQLite docs, GitHub changelogs, or direct arithmetic). No "I think" objections.

This is a phase-4 critique of the **design**, not the code. The ADR is unusually rigorous and most of its load-bearing claims hold up under verification. The findings below are the places where verification contradicts the design, exposes an untestable claim, or reveals an unhandled state/failure.

---

## Findings

| ID | Severity | Claim under critique | Cited evidence (ground truth) | Required design change |
|----|----------|----------------------|-------------------------------|------------------------|
| **F-01** | **BLOCKER** | AC-4 / §2[5] / requirements ground-truth: commits authored with the user's connected email are "verifiable on the contribution graph post-merge." The ADR treats author-email = graph attribution as settled. | GitHub graph attribution depends on the **email in the commit that lands on the parent default branch**, AND the merge method is the **maintainer's** choice, which the agent cannot control. On **squash merge**, GitHub creates "a single new commit on main"; attribution is preserved only if the squashing maintainer keeps author metadata / picks the right email — "If the person performing the merge doesn't preserve authorship metadata correctly, the commit may be incorrectly attributed to the merger, or the commit will use a non-verified email" ([GitHub squash attribution changelog 2022-09-15](https://github.blog/changelog/2022-09-15-git-commit-author-shown-when-squash-merging-a-pull-request/); [troubleshooting-missing-contributions](https://docs.github.com/en/account-and-profile/how-tos/contribution-settings/troubleshooting-missing-contributions): "Commits are only counted if they are made in the default branch"). The KPI of the whole product (visibility) can therefore silently fail on any squash-merging repo even when the PR merges. | The ADR must (a) treat graph-credit as a **separately-tracked outcome distinct from "merged"** — a merged PR that produced **no graph credit** is a partial failure the Reporter (§9 / FR-6) must surface, and (b) add a verification step that, ~24h+ post-merge, checks the parent default-branch commit's author email and the user's contribution timeline. Without this, AC-4 ("verifiable on contribution graph") cannot be demonstrated by the system itself. |
| **F-02** | **BLOCKER** | AC-4 is the only criterion that is **not observable at run time** — graph credit appears "up to 24 hours" after the merge commit lands. The ADR's state machine terminates at `merged (terminal, KPI++)` with no post-merge verification state. | "you may need to wait for up to 24 hours to see the contribution appear on your contributions graph" ([troubleshooting-missing-contributions](https://docs.github.com/en/account-and-profile/how-tos/contribution-settings/troubleshooting-missing-contributions)). The state machine §6 marks `merged` terminal immediately on detecting merge; there is no state that re-checks after the graph-propagation delay. | Add a non-terminal **`merged → graph-verify (pending ≥24h) → graph-credited | graph-missing`** transition. `graph-missing` is a recorded outcome that flags F-01-class failures. This also makes AC-4 **testable**: the test asserts the agent enters `graph-verify` and resolves it, rather than asserting against a live graph (which no CI can do — see F-09). |
| **F-03** | **MAJOR** | §2[4] / C4 / FR-3: "open a **draft PR on the fork** (base = fork default branch, head = feature branch)." The design relies on a draft PR existing *within the fork* (head and base both in the user's fork) as the approval surface. | Mechanically this is a valid **intra-repository** PR (head=`feature`, base=fork's `main`), and draft PRs are now available in **all** repos free ([draft PRs in all repositories, changelog 2025-05-01](https://github.blog/changelog/2025-05-01-draft-pull-requests-are-now-available-in-all-repositories/)). **BUT** GitHub's PR-creation UI and API default the base to the **upstream parent**, not the fork: "By default, pull requests are based on the parent repository's default branch" and "the default branch of the upstream repo seems to be taken as the base branch" with "no setting respected for PRs on a fork to change this default" ([community #11729](https://github.com/orgs/community/discussions/11729); [creating-a-pull-request-from-a-fork](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request-from-a-fork)). The agent must therefore **explicitly set base to the fork repo** in the API `create_pull` call (head and base owner both = the user), or it will accidentally open the *upstream* PR at draft time — defeating the entire approval gate (FR-3 HARD requirement). | C5 `create_draft_pr_on_fork` must be contractually specified to call `POST /repos/{user}/{fork}/pulls` (the **fork's** pulls endpoint) with `base="<fork-default>"`, `head="<feature>"`, `draft=true` — NOT the upstream pulls endpoint. Add an explicit invariant + test: "draft PR's `base.repo.full_name == head.repo.full_name == fork" — i.e., the draft PR is provably intra-fork and cannot reach upstream. This is the single most important testable guard for FR-3. |
| **F-04** | **MAJOR** | §2[4]+[5] / state machine: `draft-on-fork → approved → upstream-open`. The design implies the same draft PR transitions to upstream. | A PR's base repository is **immutable** after creation; you cannot retarget an intra-fork draft PR to the upstream repo. Publishing upstream is a **second, distinct PR** (`POST /repos/{upstream}/pulls` with `head="user:branch"`). ([creating-a-pull-request-from-a-fork](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request-from-a-fork) — head format `owner:branch` for cross-fork; base is upstream's branch.) The ADR's §5 budget and audit must account for **two PR-creation mutations per contribution** (fork draft + upstream), and the fork draft PR must be explicitly **closed** when the upstream opens, or it lingers as stale state. | State machine must show two distinct PR objects: `fork_draft_pr` and `upstream_pr`. On `approved`, the publisher (a) opens the upstream PR and (b) closes the fork draft PR with an audit event. Budget arithmetic (F-06) must count both creations. Add a state for "fork draft closed after publish." |
| **F-05** | **MAJOR** | C4 approval protocol: "Poller treats label events as the only approval source." TOCTOU: label `agent:approve-upstream` is read at poll time T1, upstream PR published at T2. The ADR has no guard for label removal, repo-archival, or policy change in the (T1, T2) window beyond F5's policy re-check. | Label state is mutable and the user (or a bot) can remove `agent:approve-upstream` between detection and publish. The design's only pre-publish re-check is **policy** (F5), not **approval freshness**. Additionally, GitHub's label-applied event carries an `actor` ([issue timeline events](https://docs.github.com/en/rest/issues/timeline), [issue-event-types](https://docs.github.com/en/rest/using-the-rest-api/issue-event-types)) — the design must verify the actor is the **user themselves**, not the agent's own token or a third party, or a mislabeled event could auto-publish. | C4 must require, immediately before publish (same transaction as the F5 policy re-check): (1) re-read the fork draft PR's **current** labels and confirm `agent:approve-upstream` is still present; (2) verify the label's `actor.login == <the user>` from the timeline event; (3) confirm the PR is still open. Make the approval re-check and policy re-check a **single atomic pre-publish gate**. Add a test for "label removed after detection, before publish → publish aborts." |
| **F-06** | **MINOR (arithmetic OK, but under-counted)** | §5: caps "≤ 8 content-creation calls/min, ≤ 50/hr"; claims "natural load (1 PR/day) sits orders of magnitude below this." | Per-contribution content-creation mutations: fork (1) + push branch (counts as git push, not REST content-creation) + fork draft PR (1) + close fork draft PR (1) + upstream PR (1) + N review replies. At 1 PR/day that is ~4–5 content-creation calls **per day**, trivially inside 80/min·500/hr platform limits ([rate-limits-for-the-rest-api](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api)) and the self-imposed 50/hr. **Arithmetic holds.** The gap is that the ADR's narrative (§2[4]) counts only **one** PR creation; F-04 adds a second creation + a close, so the ledger's category accounting must include them or the "1/day PR" budget check (which keys on `upstream_pr_opened_today`) is correct but the *content_creation* counter is undercounted by ~2/contribution. Not a limit breach; a ledger-completeness bug. | Update §5/C7 to enumerate **all** content-creation mutations per contribution (fork-create, fork-draft-PR, fork-draft-close, upstream-PR, each review reply) so the ledger is complete and the audit log (NFR-5: "every GitHub mutation logged") is actually exhaustive. The daily-PR guard stays keyed on upstream-PR only. |
| **F-07** | **BLOCKER (testability/invariant)** | §3 / §6 / delivery-plan invariant: "GitHub is the system of record, SQLite in WAL mode"; the database file lives under the project dir, which the env shows is `C:\Users\rutvi\Downloads\projects\outreach agent`. | SQLite WAL **does not work on cloud-synced / network folders**: "It is recommended not to put SQLite databases on cloud-synced folders such as Dropbox and OneDrive" and "WAL does not work over a network filesystem because WAL requires all processes to share a small amount of memory" ([sqlite.org/wal](https://sqlite.org/wal.html) / [sqlite.org/howtocorrupt](https://sqlite.org/howtocorrupt.html) / [useovernet](https://sqlite.org/useovernet.html)). A OneDrive-synced `Downloads` folder is a documented corruption vector for the exact-once budget ledger and audit log that the entire integrity story (NFR-1, NFR-5, AC-6) rests on. | The ADR must **mandate the DB path live outside any cloud-synced/OneDrive folder** (e.g., `%LOCALAPPDATA%\outreach-agent\`, which is not synced) and **fail fast at startup** if the resolved DB path is under a known sync root. Add a startup invariant check + test. Without this, the ACID guarantees the design claims (§6) are void on the target host. |
| **F-08** | **MAJOR (testability)** | delivery-plan invariant + plan step 5a: "tests mock GitHub API — never call real API in CI." §2[3] Prep Sandbox "runs the repo's own test suite + linters in a subprocess." | Two distinct testability problems the ADR does not address: **(a)** The prep sandbox executes **arbitrary third-party repo test suites** — this cannot run in CI without executing untrusted code (security + reproducibility). **(b)** The GitHubGateway (C5) is a clean mock chokepoint for *GitHub*, but the prep sandbox's subprocess execution has **no equivalent seam** — there is no `SandboxRunner` interface to mock, so the pipeline's biggest non-deterministic dependency (someone else's CI) is unmockable as designed. | (1) Add a `SandboxRunner` contract (C-new) mirroring the GitHubGateway pattern: the pipeline depends on an interface; CI injects a fake that returns canned `test_run`/`lint_run` results, so the **pipeline** is testable without running third-party code. (2) Specify that the real sandbox runs in an **isolated, network-restricted, time-bounded subprocess** (see F-10) and is exercised only in a dedicated, opt-in integration lane against **fixture repos the project owns**, never arbitrary upstream repos, never in the default CI lane. |
| **F-09** | **MAJOR (testability)** | AC-1, AC-4, AC-5 are written against "live GitHub data" / live contribution graph. The invariant says CI must never call the real API. | These two statements are in direct tension and the ADR does not reconcile them. AC-1 ("surfaces ≥10 scored candidates from **live** GitHub data"), AC-4 (graph verifiable), and AC-5 (surfaces review comments) cannot be asserted in a GitHub-mocked CI lane. | The ADR/test-strategy must split acceptance verification into: **mocked CI lane** (asserts pipeline behavior against recorded GitHub fixtures — runs every commit) and a **manual/scheduled live-smoke lane** (run by the single user, off CI, asserts AC-1/4/5 against real GitHub with the real token). State explicitly which ACs are CI-verifiable vs live-smoke-only. Until this split is documented, the test plan cannot satisfy both AC-1/4/5 and the "never call real API in CI" invariant. |
| **F-10** | **MAJOR (failure mode)** | §9 failure table omits: a repo test suite that **hangs forever**, **requires network/Docker/secrets/services**, or **fails for environmental reasons unrelated to the patch**. C3 requires `test_run.exit_code == 0` to construct a PreparedContribution. | A non-terminating or environment-dependent test suite stalls the single-threaded serialized pipeline indefinitely (no timeout policy is stated anywhere in the ADR). Many real Python/Rust/Node/React repos require Docker, a live DB, env-var secrets, or network access to run their suite — the sandbox will get `exit_code != 0` for reasons **unrelated to the patch quality**, silently dropping otherwise-good contributions and depressing throughput with no diagnostic. | Add failure mode **F8 (test-suite timeout/environment-unfit)**: every sandbox subprocess gets a hard wall-clock timeout (e.g., 15 min, config-pinned) → on timeout/non-patch-related failure, mark contribution `sandbox-unfit (reason)` (distinct from `ci-failed`, which means the patch broke tests), do not penalize repo health the same way, and surface the count in reporting. Specify network-egress restriction for the sandbox (also F-08). |
| **F-11** | **MAJOR (state machine)** | §6 state machine: transitions assume the upstream repo remains open to external PRs through the whole lifecycle. F6 covers the **approval-time** auto-close case, but not the window **after** `upstream-open`. | A repo can be **archived**, switch to members-only, or be deleted **after** the upstream PR is open and while in `review-loop`. The state machine has no transition for "upstream PR became un-actionable (repo archived / converted / deleted) mid-review." Pushing fix-ups (`review-loop → prepared'`) to an archived repo's PR will 403. | Add transitions from `upstream-open` and `review-loop` to a terminal `abandoned-upstream-unavailable (reason)` outcome, detected on 403/404/422 during monitor or fix-up push. Count it as a decided outcome for the merge-rate window (or explicitly exclude it — but decide, don't leave undefined). |
| **F-12** | **MINOR (state machine)** | §6: `rejected (terminal)` is reached via label `agent:reject` **or** PR closed. C4 says rejection reason = closing comment. | Race: the user closes the fork draft PR (rejection) at the same moment the poller has already detected `agent:approve-upstream` and begun publish (the F-05 TOCTOU, viewed from the reject side). Without the F-05 atomic pre-publish gate, the agent can publish upstream for a contribution the user just rejected-by-close. | Resolved by F-05's atomic pre-publish gate (re-confirm PR open + label present + actor == user in one transaction). Call this dependency out explicitly so it is not dropped. |
| **F-13** | **MINOR (failure mode)** | §7 LLM integration: no failure mode for **Claude API outage / timeout / rate-limit mid-prep**. §9 covers GitHub failures only. | The prep sandbox calls Claude for fix generation; an outage mid-prep leaves a contribution in a partially-prepared state. The ADR's crash-recovery (F7) keys on GitHub `intent`/`confirmed` audit pairs — it does not cover an LLM call that failed after a clone but before a diff. | Add failure mode **F9 (LLM unavailable/over-budget mid-prep)**: `prepared` is only reached on a complete, test-green patch; an LLM failure leaves the contribution in `policy-cleared` (re-enterable) with the partial work dir cleaned up, and respects the §7 monthly spend cap as a hard stop. State the LLMGateway timeout + retry policy (currently unspecified, unlike the GitHubGateway's 30s). |
| **F-14** | **MINOR (functional/Windows)** | §3: "shell out to system `git`"; clones Linux-centric repos on Windows 11. No mention of path-length or line-ending handling. | Windows default `MAX_PATH` 260-char limit breaks clones of repos with deep paths unless `core.longpaths=true`; CRLF/LF autoconversion (`core.autocrlf`) can make the agent's diff show **whole-file line-ending churn**, which is precisely the "whitespace PR" spam class the product BANS (FR-1, requirements banned-types). A line-ending-polluted diff would get the user flagged. | Pin git config for all agent clones: `core.autocrlf=false`, `core.longpaths=true`, and assert the generated diff contains **no pure line-ending changes** before constructing C3 (add to the CI-green gate). Add a test with a CRLF-containing fixture repo. |
| **F-15** | **MINOR (functional UX)** | C4 / FR-3: single-user approval via applying a GitHub **label** on the fork's draft PR, from web or mobile. | Applying a label is a multi-tap action buried in the PR sidebar on github.com mobile/web; the ADR already provides `/approve` comment as an alternative, which is lower-friction. This is a usability note, not a blocker — but the ADR lists the label as primary and the comment as parenthetical. | Minor: document both paths as first-class in C4 and confirm the poller parses the `/approve` comment with the **same actor-verification** as the label (F-05) so the comment path isn't a weaker-authenticated bypass. |

## Cross-cutting observations (not separately scored)

- **githubkit draft support is UNVERIFIED in this critique.** I could not confirm from primary source that githubkit 0.15.5's `create_pull` exposes a `draft` parameter (search returned PyGithub, not githubkit, specifics — [githubkit repo](https://github.com/yanyongyu/githubkit)). Since githubkit is auto-generated from GitHub's OpenAPI spec and the REST API's create-pull endpoint *does* accept `draft` ([community #24461](https://github.com/orgs/community/discussions/24461)), this is very likely fine — but per Rule 1 it must be **confirmed against the githubkit source/types before implementation**, not assumed. Flagging so backend-engineer verifies at implementation, not a design blocker.
- The ADR's rate-budget arithmetic, OAuth/PKCE flow, fine-grained-PAT exclusion, and review-reply endpoint correction all **verified correct** against primary sources cited in the ADR. No findings there. Praise omitted per gate rules — recording only that these dimensions passed.

## GATE VERDICT: **PASS-WITH-CONDITIONS**

The design is fundamentally sound and well-grounded; it does **not** go back to a blank-slate redesign. But three **BLOCKER**s touch acceptance criteria and a core integrity guarantee and MUST be resolved in the ADR before implementation begins:

1. **F-07** — DB must not live on a OneDrive-synced path; ACID/exactly-once guarantees (AC-6, NFR-1/5) are void otherwise. *(cited: sqlite.org)*
2. **F-01 + F-02** — AC-4 (contribution-graph credit) is neither modeled nor verified; squash-merge can silently zero the product's entire KPI, and the only non-run-time-observable criterion has no post-merge verification state. *(cited: GitHub squash changelog + troubleshooting-missing-contributions)*
3. **F-03** — the approval gate (FR-3, HARD requirement) depends on the draft PR being provably **intra-fork**; the GitHub default base = upstream parent means a naive `create_pull` would open the upstream PR at draft time and bypass the gate entirely. *(cited: GitHub fork-PR docs + community #11729)*

Additionally, the MAJOR findings F-04, F-05, F-08, F-09, F-10, F-11 must be **answered in writing** in the revised ADR (added states/contracts/failure modes), since each is backed by primary-source evidence of a real gap. MINORs may be tracked as implementation conditions.

**Path forward:** solution-architect revises ADR-001 to address the three BLOCKERs and six MAJORs above with the cited required changes, then re-submits to this gate. The pipeline does **not** proceed to backend-engineer (chain step 3) until the BLOCKERs are closed.
